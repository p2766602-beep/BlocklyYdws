> 2026-07-22對話定案，取代`prompts/密室逃脫檢核-GEM.md`裡原始單一prompt設計中「關卡狀態/金鑰
> 完全靠AI自律」的部分。原始文件的遊戲設計本體（5關結構、蘇格拉底式引導、AI協作學習回顧）維持
> 不變，本文件只補齊工程架構層面的決策。

## 1. 定位與範圍

- **獨立於BlocklyYdws之外的新專案**，比照現有`blockly-lab`／`YDWS-CodingBank`／`BlocklyYdws`
  三專案並存模式（各自獨立git歷史，透過共用`專案規劃摘要.md`＋各自`CLAUDE.md`互相提醒）。
- 核心內容：通用演算法密室，**資優班課程定位**（2026-07-23確認），13個主題、學生自選主題、
  不綁定任何Blockly課程或task。主題與建議學習順序（軟性推薦路徑，見§8，不鎖關）：

  | 階段 | 主題 |
  |---|---|
  | Tier1 基礎・陣列線性掃描 | 1.找最大值　2.找最小值　3.線性搜尋 |
  | Tier2 排序三部曲 | 4.氣泡排序　5.選擇排序　6.插入排序 |
  | Tier3 分治初體驗 | 7.二分搜尋 |
  | Tier3.5 遞迴橋樑 | 8.遞迴基礎　9.合併排序 |
  | Tier4 演算法設計策略／圖走訪 | 10.貪婪演算法　11.DFS　12.BFS |
  | Tier5 總整理 | 13.動態規劃 |

  原案是11主題（無遞迴基礎、無合併排序），2026-07-23因確認是資優班課程、不需要保留給一般班
  的簡化版路線，決定全部納入，不設選修分支。設計邏輯：每個Tier的技巧會被下一個Tier重新利用或
  正面對照（例如選擇排序=重複執行找最小值；二分搜尋的前提是先學過排序；合併排序回頭用剛學的
  遞迴重新解排序問題；動態規劃建議以「貪婪在此題會選錯」的題目切入，跟第10關正面對照）。
- **情境C（AI伴學-Blockly解題陪伴）與情境D（SRC助教）不在此文件討論範圍**——情境C的跳關規則
  （學習概念連結不可跳過，其餘階段可依學生明說跳過/直達除錯）是`docs/AI伴學GEM-V62-AI素養
  評估版.md`自己的調整，跟本專案無關。情境D的定位見下一節。

## 2. 跟主平台的邊界：情境D（SRC助教）怎麼處理

**決定：SRC00維持在BlocklyYdws主平台的AI伴學浮動面板內**（用courseCode路由），不搬到獨立專案。
但情境D本質上是「同一套密室逃脫引擎，換SRC知識包」（`SRC_GPT_Minimal_Knowledge_V3.md`取代通用
演算法主題），所以：

- **引擎程式碼只寫一份，放在這個新獨立專案裡**，不要為了SRC00而在`workers/ai-companion`（情境C
  用的簡單聊天Worker）裡另外重寫一套關卡狀態機邏輯，避免簡單聊天Worker背上不必要的複雜度。
- **整合機制建議：比照`src/smartring/simulator-bridge.js`現有的iframe + postMessage嵌入模式**
  （`getBasePath()` + `iframe.contentWindow.postMessage(...)`，這套手法已經在SmartRing模擬器上
  驗證過）。主平台偵測到courseCode為SRC00時，AI伴學面板改嵌入獨立密室逃脫專案部署好的
  `embed.html`（帶入知識包參數＝SRC、學生身分參數），而不是继续走`aiCompanion.js`現有的純文字
  chat fetch流程。
- 學生身分（班級/座號/姓名）透過URL query string或postMessage傳入embed頁面，做法比照
  `simulator-bridge.js`。
- 好處：情境C的Worker維持簡單無狀態、密室引擎的複雜狀態邏輯只存在一個地方、兩邊各自能獨立部署
  迭代。

## 3. 遊戲設計（沿用既有GEM文件，僅列本次調整）

- 5關結構（現象推論/選項辨析/邏輯除錯/因果重組/知識遷移）、每主題碎片拆解、最終合併金句、
  AI協作學習回顧（8.2~8.4節）維持`密室逃脫檢核-GEM.md`原設計不變。
- **金鑰定位澄清**：純粹是遊戲儀式感/學習動機工具（過關後截圖跟主持人領獎品），**不是安全憑證**，
  不需要密碼學等級的不可預測性。

## 4. 答案判定架構——按鈕與文字雙軌並存（重要修正）

前一版提案曾誤把Level 2/4/5的答案判定完全交給後端deterministic比對、排除AI判斷，**這是錯的**，
已修正為混合式：

- **Level 2（選項辨析）／Level 4（因果重組）／Level 5（知識遷移危機）**：AI以結構化JSON回傳
  `{ scenario, options: [...], correctIndex, hint }`（Gemini API的`responseSchema`結構化輸出，
  不需換供應商）。前端渲染成按鈕（Level 2/5）或拖曳排序（Level 4），**這是快速路徑**：
  學生點選/拖放後，答案由**後端程式碼直接比對`correctIndex`**，只請AI生成對應的鼓勵/引導回饋文字，
  不重新用AI判斷對錯。
- **但文字輸入框永遠保留、永遠可用**：學生不見得會乖乖點按鈕——可能改用自己的話講推理過程、
  問「這個詞是什麼意思」、表達卡關或部分理解。**只要學生選擇打字而非點選，一律進AI判斷通道**，
  AI要理解內容並給予適切回饋或引導，跟現在情境C的自由對話行為一致。按鈕只是「多一條快速通道」，
  不是取代文字對話的判斷責任。
- **Level 1（現象推論）／Level 3（邏輯除錯）**：本質是開放式簡答，維持純文字輸入＋AI判斷，
  不做按鈕化，以保留「自己說出核心概念」的教學意義。

## 5. 後端狀態追蹤

- 沿用`workers/ai-companion/src/index.js`的KV「讀取→+1→寫回」計數器手法，但改記錄
  `(studentId, topic)`目前關卡序號、已收集碎片內容、是否已達終局，取代目前情境C「完全靠前端
  echo回來的對話history」的做法。
- **理由**：情境C沒有實體獎勵，前端持有全部state風險可接受；密室逃脫掛實體獎品後，學生可用
  devtools竄改自己送出的history騙AI直接吐出「看起來像過關」的金鑰去截圖交差，所以後端要有一份
  自己的、學生無法竄改的關卡進度紀錄。
- 金鑰在後端偵測到「已達終局」狀態時，由伺服器自己隨機挑一個三位數質數產生，不讓AI自己編
  （避免AI編出不是質數、或重複配發的金鑰）。

## 6. 視覺呈現方向

- 結構化JSON生成內容 + 前端渲染卡片/按鈕/拖曳目標，不是直接把AI寫的整段文字印成聊天泡泡。
- 每種題型（選項辨析／因果重組／知識遷移危機……）各自預先設計一套固定插畫/icon風格，套用到不同
  主題的文字內容上；**不做即時AI生圖**（成本、延遲、風格不一致、對兒童的圖像安全疑慮都是問題）。
- 通關進度視覺化：碎片收集/開鎖動畫，可參考SmartRing模擬器LED圓環的視覺語言（若SRC00沿用同一
  套引擎，這個視覺母題也能呼應SmartRing主題）。
- 金鑰揭曉畫面獨立設計成適合截圖分享的「戰利品卡」元件（證書/獎盃風格），因為screenshot本身就是
  這個機制的實際產出物，值得專門設計而不是埋在對話紀錄裡。
- 答對/答錯給輕量CSS動畫回饋（碎片飛入收集區、卡片震動），成本低但遊戲感提升明顯。

## 8. 遊戲化總覽（軟性推薦路徑，2026-07-23定案並實作完成）

使用者提供RPG世界地圖風格參考圖，評估後定案，並在同一輪完成實作＋瀏覽器端對端驗證：

- **軟性推薦路徑，不做硬鎖關**：13主題（§1）依Tier順序在總覽畫面上顯示建議路徑＋已完成進度，
  但任何節點都可自由點擊進入，不因未完成前一主題而鎖住。理由：與「學生自選主題、不綁定」的
  設計初衷相容，資優班學生想跳關挑戰BFS/DP也不受限制。
- **範圍限定MVP，美術地圖化延後**：**不做**像參考圖那樣的像素風RPG地圖插畫——美術素材授權/
  成本問題，改用純CSS/SVG節點圖。
- **與主平台的連結方式**：跟情境D（SRC00，§2）的iframe+postMessage嵌入模式**刻意不同**——
  密室逃脫是獨立產品體驗，從BlocklyYdws連過去要用一般`<a target="_blank" rel="noopener">`
  開新分頁/新視窗，不做浮動視窗或iframe嵌入（那套只保留給情境D）。
- **桌機/平板優先，非手機版面**：使用者實測發現原本沿用垂直切片的手機單欄卡片版型（`#app`
  寬度上限640px、字級普遍14-16px）不適合密室逃脫實際會使用的桌機/平板場景。總覽畫面獨立採用
  寬版面（`max-width:1180px`），其餘畫面（entry/level/completed）字級與卡片寬度也一併調高
  （640→720px，h1 20→24px、label 14→15px、input/button字級同步微調）。

### 8.1 節點排版（已實作）

「捷運路線圖」式橫向時間軸：6欄對應Tier1／Tier2／Tier3／Tier3.5／Tier4／Tier5，一條水平主幹線
貫穿全部欄位的垂直中心，欄內有多個節點時（Tier2/Tier3.5/Tier4）用一條垂直支線把該欄節點群跟
主幹線連起來，純CSS絕對定位實現，不需要SVG動態算座標或圖表函式庫。每個節點是圓形按鈕，
icon（emoji）+主題名稱，四種視覺狀態：

| 狀態 | 樣式 | 判定 |
|---|---|---|
| 已完成 | 綠框＋✅徽章 | 該主題5關全破（見8.2） |
| 進行中 | 黃框＋◐徽章 | 已開始但未破完5關 |
| 建議下一步 | 藍框＋脈動發光動畫 | 依Tier順序，第一個「還沒完成」的主題 |
| 可挑戰但非建議 | 灰階弱化 | 其餘未完成主題 |

### 8.2 已完成判定標準（使用者2026-07-23拍板）

**5關全破才算✅完成**，「進行中」（部分過關但未破完5關）用另一個視覺狀態表示，不會被算作完成
——避免推薦路徑的完成度標示失去意義。

### 8.3 難度選擇時機（使用者2026-07-23拍板）

**難度在entry畫面選一次，整個總覽旅程（13主題）共用同一個難度場次**，不支援每個主題各自選
不同難度。理由：學生進入密室逃脫時的難度定位應該是一致的體驗設定，不是逐關切換的參數。

### 8.4 狀態機（已實作）

`entry`（座號/姓名/難度，只填一次）→ `overview`（讀取13主題進度，點節點進入）→ `level`（原有
5關流程不變）→ 過關後回到`overview`（不是舊版「完全重填identity」的`completed`終點迴圈），
總覽的`completed`畫面本身維持不變（金鑰＋AI協作學習回顧卡片），只是按鈕從「再挑戰新的密室」
改成「返回總覽」，且不清空`studentId`/`difficulty`。另外總覽畫面加一個「換人/重新開始」連結，
供教室共用裝置換學生使用時完全重置。

### 8.5 後端整合（已實作）

新增`POST /overview`端點（`workers/escape-engine/src/index.js`），輸入`{studentId, difficulty}`，
對13個已知主題代碼各自查一次既有的`(topic, difficulty, studentId)`KV進度（§5既有儲存格式，
非新格式），回傳每個主題的`not_started`/`in_progress`/`completed`狀態＋目前關卡數。因為難度
在entry固定、13主題代碼是靜態已知清單，**用13次平行KV get就夠，不需要KV `list(prefix)`**，
避免list()的最終一致性風險。
